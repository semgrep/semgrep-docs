# Tokens

An access token is a secure credential used to authorize requests to Semgrep AppSec Platform or the Semgrep API without a username and password. Each token is associated with a specific Semgrep account and has a defined set of [scopes](#token-scopes), which define the permissions granted to the bearer of the token.

Semgrep supports the following types of tokens:

- API tokens
- CLI tokens
- Semgrep service tokens

## API tokens

If you are an admin for your Semgrep deployment, you can create and manage API tokens in Semgrep AppSec Platform. These are used for API access, integrations, and automation. 

To create an API token:

1. Sign in to [<i class="fas fa-external-link fa-xs"></i> Semgrep AppSec Platform](https://semgrep.dev/login).
2. Go to [**Settings > Tokens > API tokens**](https://semgrep.dev/orgs/-/settings/tokens/api).
3. Click **Create new token**.
4. Copy the **Secrets name** and the **Secrets value**, and save these values. The **Secrets value** is your token and only shown at this time.
5. Select the **Token scopes**.
6. Optional: change the **Name** of the token. This is the value used in the list of tokens associated with your Semgrep deployment.
7. Click **Save** to proceed.

For auditing purposes, API tokens are associated with the user who created them. However, they remain valid until manually revoked, even if the creator is no longer associated with the deployment. Rotate regularly and revoke during admin offboarding.

## CLI tokens 

CLI tokens authenticate users running scans or publishing rules from the Semgrep CLI. Once you've [set up the Semgrep CLI](/getting-started/cli#set-up-semgrep), you can create a CLI token by running the following command:

```bash
semgrep login
```

Both members and admins of a deployment can create CLI tokens. The CLI token allows users can run scans on their local machine using the `semgrep ci` command. This sends findings data to Semgrep AppSec Platform. It also allows users to [publish rules](/writing-rules/private-rules#creating-private-rules) using `semgrep publish`.

The permissions assigned to a CLI token cannot be elevated. To access the Semgrep API, users must be elevated to the **admin** role, if they're a **member**, and create an API token.

For auditing purposes, Semgrep [records the user that generated the CLI token](https://semgrep.dev/orgs/-/settings/tokens/cli), but the user's actions are attributed to the token, *not* the user.

Logging out of the Semgrep CLI took using `semgrep logout` removes the local token, but this does not invalidate the token.

## Semgrep service tokens 

Semgrep service tokens are functionally the same as API tokens, but instead of being manually generated by a user, they are automatically generated during repository onboarding for CI/CD scans. These tokens authenticate agents running automated scans within CI environments. The default scope of these tokens is Agent/CI, but you can grant them the API scope as well. 

# Token scopes

The following table displays token scopes and their permissions:

| Token scope | Send findings from a remote repository | Send findings from a local repository | Send PR or MR comments | Connect to Semgrep API |
| - | - | - | - | - |
| API | ❌&nbsp;No | ❌&nbsp;No | ✔️&nbsp;Yes | ✔️&nbsp;Yes |
| CLI | ❌&nbsp;No | ✔️&nbsp;Yes | ❌&nbsp;No | ❌&nbsp;No |
| Service (CI) | ✔️&nbsp;Yes | ✔️&nbsp;Yes | ✔️&nbsp;Yes | ❌&nbsp;No |

The following table displays typical uses for token scopes:

| Token scope | Typical uses |
| - | - |
| API | Used to access Semgrep's API. |
| CLI | Auto-generated by Semgrep CLI when a member is logging in through Semgrep CLI. Use this scope to scan your code locally using your organization's configured Policies, including private rules. The permissions of these tokens cannot be escalated. |
| Service (CI) | Generated by Semgrep AppSec Platform when onboarding (adding) a repository to Semgrep AppSec Platform. For non-GitHub Actions users, you may have to copy and paste the token value into your CI provider's interface. |
