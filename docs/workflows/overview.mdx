Semgrep Agentic Workflows encodes multi-step application security analysis as reproducible pipelines. A single agentic workflow can run several analysis steps in sequence. An agentic workflow takes code and security context as input, combines program analysis, deterministic tools, and constrained AI-based reasoning, and produces reviewable security issues.

During the beta, Semgrep Agentic Workflows provides a curated set of vulnerability **detection** workflows that you run from **Semgrep AppSec Platform**. Creating, modifying, or publishing custom agentic workflows is not supported. Local CLI execution is not available; Agentic Workflows run as a Semgrep-hosted service.

To run your first agentic workflow, see [Get started](/workflows/get-started).

## Agentic Workflows versus other Semgrep analysis

Semgrep Agentic Workflows performs targeted analysis that combines multiple analysis techniques and reasoning steps. It complements Semgrep Code's rule-based scanning and AI-powered detection.

The following table summarizes when to use each type of analysis.

| Analysis | Primary use | How it works | Output |
| --- | --- | --- | --- |
| [Semgrep Code](/semgrep-code/overview) rule-based scanning | Continuously identify code that matches known insecure patterns | Applies deterministic rules to source code | Code findings |
| [AI-powered detection scans](/semgrep-code/ai-powered-detection-concepts) | Find business logic flaws such as IDORs and broken authorization that are hard to express as generic rules | Uses AI-based analysis together with Semgrep program analysis | AI-powered detection findings |
| [Semgrep Multimodal](/semgrep-multimodal/overview) | Help prioritize and remediate existing findings | Applies AI to Semgrep findings and related context | Triage and remediation guidance |
| Semgrep Agentic Workflows | Combine multiple analysis techniques and execute them stepwise across repositories | Runs a built-in sequence of analysis steps, such as specialized analyzers and AI reasoning, against selected repositories | Agentic Workflows issues |

## How Agentic Workflows work

The following concepts describe how analysis is organized:

- **Agentic workflow**: A reusable definition of an application security pipeline. It specifies the analysis to perform and the type of issue or coverage objective to evaluate.
- **Run**: One execution of an agentic workflow against the repositories selected by a user.
- **Job**: An individual unit of work within a run. A job can perform program analysis, run a deterministic tool, apply AI-based reasoning, or review intermediate output.
- **Finding**: The immutable raw output of a detection workflow. Findings appear on the job details page after a run finishes.
- **Issue**: The deduplicated, processed output of the pipeline. Issues appear on the **Agentic Workflows** page in Semgrep AppSec Platform and include the evidence and context available from the run.

When you start an agentic workflow, Semgrep accesses the selected repository through its source code manager connection. The workflow runs its jobs and produces findings. Semgrep processes and deduplicates those findings into reviewable issues. You can monitor the run and review its issues in the **Agentic Workflows** page in Semgrep AppSec Platform.

## Detection coverage

During the beta, Semgrep offers nine pre-built detection workflows. Together, these workflows cover 70+ [CWEs](https://cwe.mitre.org/), including common injection, XSS, SSRF, and cryptography issues, as well as broader web, API, and crypto security analysis. This coverage is **additional** to what [Semgrep Code](/semgrep-code/overview) and [Semgrep Pro Engine](/semgrep-code/semgrep-pro-engine-intro) already cover.

**Semgrep's default** runs all nine of these workflows.

Coverage depends on the workflow, repository language, framework, and available code context. A workflow might not analyze every file or detect every instance of a vulnerability class. Continue to use Semgrep Code and Semgrep Supply Chain for continuous rule-based scanning.

### Available workflows

| Workflow | What it finds |
| --- | --- |
| SQL injection | Traces user-controlled data into dynamic queries that lack effective parameter binding or allow-list validation. |
| XSS | Traces user-controlled input to unsafe HTML rendering sinks, across reflected, stored, and DOM-based cases. |
| SSRF | Analyzes attacker control, validation, and bypass paths using repository-specific rules. |
| Command injection | Traces untrusted input to shell and process-execution APIs across supported languages. |
| NoSQL injection | Traces untrusted input into query and scripting operations in MongoDB, Redis, and Elasticsearch. |
| Crypto weak hash | Flags security-sensitive use of MD2, MD4, MD5, and SHA-1, while filtering out benign checksum, cache-key, and identifier uses. |
| Broad web security | Audits application code across web, API, cryptography, access control, and AI security, then filters by confidence and exploitability. |
| Broad API security | Audits REST, GraphQL, gRPC, SOAP, JSON-RPC, and webhook APIs, then prioritizes findings by exploitability. |
| Broad crypto security | Audits cryptography, password hashing, key management, signatures, randomness, and TLS, then prioritizes by exploitability. |

## Credit usage and billing

Agentic workflow runs consume [AI credits](/usage-and-billing/overview#ai-credits). Credit use is variable per run and depends on compute, AI tokens, repository size, and complexity. Agentic Workflows typically use more credits than [AI-powered detection](/semgrep-code/ai-powered-detection-concepts) scans because they perform broader, multi-step analysis. For a large repository, running the full recommended pack can cost $100 or more.

Before expanding Agentic Workflows to more repositories, review usage in **[usage and billing UI location]** and confirm that your organization has enough credits for the planned rollout. Start with a small set of important projects so you can understand credit consumption before scaling.

## Beta limitations

During the beta:

- Runs must be started on demand. Scheduled runs are not available.
- Agentic Workflows runs only as a Semgrep-hosted service and cannot be run in customer CI environments.
- Agentic Workflows issues are managed separately from findings in other Semgrep products.
- The API is experimental and can change.
- BYOK, MDR or ZDR, and Network Broker configurations are not supported.
- Detection coverage and supported repositories vary by workflow.

Semgrep plans to refine Agentic Workflows throughout the beta based on customer feedback.

## What's next

Detection is one stage of a longer loop that also includes triage and remediation. Semgrep is developing additional workflow types on the same platform, including:

- **Triage workflows**: In early testing, a triage workflow reached 96% agreement with security analysts.
- **Remediation workflows**: An Autofix workflow saved roughly 30 minutes per finding.

Expect continued investment in triage and remediation workflows, and in the context that improves accuracy across all workflows.
